# Campus Match AI 对话与项目推进总结

更新时间：2026-06-13

## 1. 项目目标

本轮对话围绕一个校园社交匹配 Demo 展开，目标是把“心动星球 / 闪电搭子 / 星图管理员台”做成能演示、能解释、能和课程技术对应上的项目。

核心要求包括：

- 用户端要像真实手机 App，而不是一堆技术说明。
- 管理员端要能展示用了哪些算法、数据结构和工程模块。
- 技术部分不能只写概念，要能说明具体怎么做、用了什么模型、输入输出是什么、结论是什么。
- GitHub Pages 要能给组员和老师直接打开。
- 本地版要保留更完整的实时 API / RAG 演示能力。

## 4. 已实现或准备好的课程技术点

### 4.1 向量检索与 RAG

项目中已使用向量检索和 RAG 思路：

- 把用户画像、匹配理由、破冰问题、安全规则等文本组织成可检索知识。
- 用户输入后进行 Top-K 检索。
- 根据检索结果生成聊天建议或回复。
- 前端展示检索命中文档、分数、来源和推理路径。

相关课程点：

- 向量数据库
- Embedding
- Top-K 检索
- RAG
- 检索增强生成

### 4.2 FAISS Flat / IVF / HNSW 对比

已经补充 ANN 检索实验，用来解释：

- Flat 检索：暴力相似度计算，准确但慢。
- IVF：先聚类再查倒排桶，速度更快但可能损失召回。
- HNSW：图结构近邻搜索，适合低延迟近似检索。

前端管理员端展示：

- 不同索引的召回结果
- 检索耗时
- 索引结构差异
- 为什么实际系统不会只用 Flat

### 4.3 BM25 + 向量混合检索

项目中加入 BM25 Hybrid 思路：

- BM25 适合关键词明确的查询，如“185”“羽毛球”“金融男”。
- 向量检索适合语义模糊查询，如“想找会弹吉他的帅哥”。
- 混合检索能同时覆盖硬条件和语义相似。

### 4.4 知识图谱 / Neo4j

已经生成 Neo4j 所需的 CSV 和 Cypher 文件，并在前端管理员端做图谱展示。

图谱包含：

- 用户节点
- 兴趣节点
- 标签节点
- 场景节点
- 匹配关系
- 风控关系
- 治理事件

可讲解逻辑：

- 用户不是孤立样本，而是图中的节点。
- 兴趣、校区、时间、风控事件都是可连接实体。
- 推荐、风控、寻人启事都可以转成图查询问题。

### 4.5 GNN / 图特征

已做过 GNN 相关实验与说明，定位为：

- 用图结构预测“用户 A 和用户 B 是否可能匹配”。
- 图中的边和节点特征来自项目合成数据、用户画像、标签、互动记录和治理记录。
- 适合答辩时解释“为什么图结构比单纯标签匹配更强”。

当前更偏 Demo 级实验，不是线上大规模生产训练。

### 4.6 规则风控与信用体系

项目包含星际信用和风险规则：

- 爽约 / 迟到 / 不安全反馈会影响推荐可见度。
- 信用分低的用户会被降权或进入人工复核。
- 闪电搭子场景中，线下核销和 LBS 验证被设计为闭环。

   - 导入 CSV。

4. 完善 GNN 实验讲解
   - 明确训练数据来源。
   - 明确正负样本构造方式。
   - 明确模型输入特征。

   - 当前 `demo/index.html` 内嵌大量数据，文件较大。
   - 后续可以把数据拆成 JSON 文件，提高加载速度和维护性。

## 9. 关键结论

这个项目目前不是只做了静态页面，而是已经形成了三层结构：

- 用户体验层：手机端星球、雷达、消息、档案。
- 算法展示层：RAG、向量检索、ANN、BM25、图谱、GNN、风控。
- 工程交付层：GitHub Pages 静态版、本地 API 版、文档和 PPT 讲稿。

答辩时建议不要讲成“我们调用了大模型”，而是讲：

> 我们把校园社交中的画像、匹配、聊天、线下邀约和风控都转成了可检索、可解释、可视化的结构化流程。用户端看到的是自然的星球互动，管理员端能看到背后的检索链路、图谱关系、风控规则和模型实验。

这样更容易体现项目完整度和课程技术结合度。
