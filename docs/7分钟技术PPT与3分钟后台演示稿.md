# 7 分钟技术 PPT 与 3 分钟管理员后台演示稿

更新时间：2026-06-15

## 0. 语速与总字数控制

公开资料给出的英文公开演讲语速通常在 100-165 words/min 之间：

- Microsoft 演讲者指导建议公开演讲大约 100-165 words/min。
- VirtualSpeech 给出的 presentation 舒适区间是 100-150 words/min。
- Debatrix 的演讲计时器认为英语演讲常见值约 140 words/min，慢速 110，快速 170。

中文技术汇报不能直接按英文单词数折算。我们这次有算法名、图、表格和后台演示，应该留停顿。建议按 **220-250 个汉字/分钟** 准备。

因此：

- 7 分钟 PPT：约 1540-1750 个汉字。
- 3 分钟后台演示：约 660-750 个汉字。
- 总体口径：宁可少讲一点，也要讲清楚“为什么用、怎么做、输出在哪里看”。

## 1. 7 分钟 PPT 总结构

建议做 **7 页 PPT**，每页 40-75 秒。不要把所有技术名堆出来，而是用一条主线讲：

> 用户原本给我们的信息是自然语言、兴趣、聊天、位置和反馈。我们把它们转成结构化画像，再进入向量检索和知识图谱，最后通过 RAG、图学习和治理规则变成可解释推荐、聊天辅助和安全管理。

## 2. 每页 PPT 具体讲法

### 第 1 页：从校园社交问题到知识增强系统

页面放什么：

- 一张总流程图：访谈/行为输入 -> 画像抽取 -> 向量检索 -> 知识图谱 -> GraphRAG -> 匹配/聊天/风控。
- 右侧放三个产品入口：心动星球、闪电搭子、管理员星图台。

讲稿：

> 我们做的不是一个单纯的社交页面，而是把校园社交里的模糊信息变成可以计算、可以解释、可以治理的知识系统。用户说“想找人吃饭”“想慢慢了解”“今天有点累”，这些原本只是自然语言。我们的处理链路是：先抽取成结构化画像，再转成向量和图关系，然后通过检索、排序、RAG 和风控策略，输出推荐、破冰回复、搭子候选和安全提醒。用户端看到的是自然的 App，管理员端看到的是背后的技术证据。

课程对应自然说法：

- 这里不是硬套某一章，而是把知识管理里的“采集、组织、检索、应用、更新”做成完整产品链路。

### 第 2 页：画像不是标签堆砌：访谈抽取与本体校验

页面放什么：

- 左侧：4 轮 AI 访谈样例。
- 中间：实体抽取结果，如 Interest、Value、Goal、CommunicationStyle。
- 右侧：三元组和本体校验结果。

讲稿：

> 注册后的画像不是手动写死标签。我们先把模拟访谈拆成问答文本，再抽取兴趣、价值观、关系目标、沟通方式、时间偏好、雷点和校区。比如“我喜欢古典音乐，也比较看重边界感”，会变成 User-LIKES-古典音乐、User-VALUES-边界感。接着我们用本体约束校验这些三元组，检查 User 能不能连这个关系、object 是否真的来自用户字段。这样做的意义是，后续检索和图谱都有来源，老师问“标签从哪里来”，我们可以直接给出访谈证据和校验结果。

关键参数/结果：

- 120 个用户。
- 画像三元组约 3365 条。
- 本体校验通过率 1.0。
- API 样例里访谈抽取 10 条，均返回可解析实体和三元组。

课程对应自然说法：

- 这部分对应知识图谱构建中的知识抽取、本体建模和三元组质量控制。

### 第 3 页：搜索不是筛选框：向量、BM25、FAISS

页面放什么：

- 示例 query：“想找个会弹吉他的阳光学长”。
- 展示四路召回：BM25、Dense Vector、Graph Terms、Hard Constraints。
- 下方展示 FAISS Flat / IVF / HNSW 对比表。

讲稿：

> 雷达搜索不能只靠关键词。用户说“会弹吉他的帅哥”，如果只做关键词，可能漏掉“音乐技能”“艺术细胞”；如果只做向量，又可能把硬条件不满足的人排上来。所以我们做了混合检索。BM25 负责精确词命中，Dense 向量负责语义相似，图谱约束负责把“吉他”扩展到音乐和艺术相关画像，硬条件负责性别、时间、地点等约束。文本向量使用 384 维句向量，检索侧做了 FAISS Flat、IVF、HNSW 对比。Flat 更准确但像暴力搜索，IVF 和 HNSW 更接近真实系统里的近似最近邻检索。

关键参数/结果：

- 文本向量维度：384。
- BM25 参数：k1=1.5，b=0.75。
- 融合权重：BM25、Dense、Graph、Constraint 四路加权。
- 管理员端可展示 Top-K、分数、召回来源和 ANN 耗时。

课程对应自然说法：

- 这里自然对应向量数据库里的 embedding、Top-K、ANN 索引和混合检索。

### 第 4 页：推荐为什么可信：Neo4j 与 GraphRAG

页面放什么：

- Neo4j 产品图截图：用户、兴趣、价值观、推荐、搭子、热度、首约、安全、治理。
- 一条推荐路径：U001 -> 剧本杀/边界感/认真了解 -> U010。
- 推荐理由示例。

讲稿：

> 推荐不能只给一个分数，否则用户和老师都不知道为什么。我们把用户、兴趣、价值观、目标、搭子任务、聊天热度和信用治理都写进产品图谱。图谱不只是展示，它会参与解释：两个用户为什么匹配，可以从共同兴趣、共同价值观、沟通风格、空闲时间等路径找证据。GraphRAG 的做法是先从图里取多跳路径，再把路径组织成自然语言理由，例如“你们都喜欢剧本杀，也都重视边界感，所以适合从低压力破冰开始”。这样推荐不是黑盒，而是能追溯到图上的节点和关系。

关键参数/结果：

- Neo4j 产品图：约 5209 节点、14600 关系。
- 覆盖画像、推荐、GraphRAG 证据、闪电搭子、关系热度、首约安全、信用治理。

课程对应自然说法：

- 这里把知识图谱、图数据库和 GraphRAG 放在同一个产品链路里。

### 第 5 页：排序如何学习：图算法、GraphSAGE、GCN

页面放什么：

- 推荐分数拆解条形图：文本语义、图谱相似、价值观、GNN link、图片。
- GraphSAGE 链接预测样例。
- GCN 风险节点分类样例。

讲稿：

> 最终推荐不是单一路径决定，而是多路特征融合。我们用文本相似度表示语义接近，用图谱相似度表示共同邻居和路径证据，用价值观和目标匹配表示关系稳定性，再加入 GraphSAGE 链接预测分。GraphSAGE 的任务可以理解为：在用户-属性图上预测两个用户未来是否可能形成匹配边。除此之外，风险治理部分做了 GCN 风险节点分类，用用户邻域、信用事件和行为标签判断是否需要降权或复核。这样一来，图计算不只用于好看，而是进入推荐排序和风控策略。

关键参数/结果：

- 推荐页展示 final score 与各子分数。
- GNN link score 已接入推荐排序。
- GCN 风险分类用于解释信用治理和可见度降权。

课程对应自然说法：

- 对应图计算里的中心性、共同邻居、链接预测和节点分类。

### 第 6 页：聊天为什么不重复：实时 RAG 与安全校验

页面放什么：

- RAG 流水线图：用户消息 -> query rewrite -> router -> Top-K retrieval -> compression -> answer -> safety verifier。
- 一条 API Trace：用户消息、Top-K 证据、最终回复、安全校验。

讲稿：

> 早期聊天如果只靠模板，会反复出现同一句话，比如一直问剧本杀。现在我们把聊天改成 RAG 流水线：用户输入一句话后，系统先改写 query，再根据意图选择画像上下文、破冰库、首约安全库或场景库，随后检索 Top-K 文档，压缩证据，再生成回复。最后还有安全校验，避免编造不存在的活动，或者在首约时忽略公开地点和时间边界。我们也做了少量 API 调用证明链路可接大模型，但核心检索、路由、证据和安全校验都是本地可复查的。

关键参数/结果：

- 访谈 API Trace：10/10。
- RAG 回复 API Trace：10/10。
- GitHub Pages 展示静态 trace，本地 API 版支持实时 `/api/chat-rag`。

课程对应自然说法：

- 这里自然对应知识增强 Prompt 和 RAG，不是“直接问大模型”，而是先检索证据再生成。

### 第 7 页：从推荐到运营：搭子、热度、信用治理

页面放什么：

- 闪电搭子卡片：任务、时间、地点、安全等级、候选搭子。
- 关系热度曲线。
- 信用治理策略：降权、冷却、复核。

讲稿：

> 最后一层是动态更新。心动星球偏长期匹配，闪电搭子偏即时任务，所以它会更看重时间窗口、地点距离、任务类型和安全上下文。聊天后，关系热度会根据消息数、回复延迟、积极比例和共同话题变化；线下邀约会结合地点人流、公开程度和信用记录做提醒；如果用户多次爽约或被不安全反馈，推荐可见度会下降。总结来说，我们把课程里的知识抽取、向量检索、图谱、RAG 和图学习，落到了一个完整的校园社交闭环里。

课程对应自然说法：

- 这里对应动态知识管理和知识治理：知识不是一次生成后不变，而是会随互动反馈更新。

## 3. 7 分钟 PPT 的压缩版口播稿

如果你想背一版连续稿，可以用下面这版，控制在约 7 分钟：

> 我们这个项目不是单纯做一个校园社交界面，而是把校园社交中的自然语言、兴趣、聊天、位置和反馈，变成可以检索、可以解释、可以治理的知识系统。整体链路是：先做 AI 访谈和画像抽取，再进入向量检索和知识图谱，然后用 GraphRAG、图学习和风控策略输出推荐、破冰回复、闪电搭子和安全提醒。
>
> 第一层是画像生成。用户注册后会有轻量访谈，我们从问答文本里抽取兴趣、价值观、关系目标、沟通方式、时间偏好和雷点，生成 User-LIKES-Interest、User-VALUES-Value 这类三元组。为了避免图谱乱连，我们定义了 Ontology，本体会检查关系类型和字段来源。本次生成约 3365 条画像三元组，本体校验通过率是 1.0，所以后面的推荐和检索有可追溯来源。
>
> 第二层是检索。雷达搜索不是普通筛选框。比如用户说“想找会弹吉他的阳光学长”，系统会同时跑 BM25、384 维语义向量、图谱约束和硬条件约束。BM25 保证关键词命中，向量保证语义泛化，图谱把吉他扩展到音乐和艺术相关画像，硬条件保证性别、时间、地点不被忽略。我们还做了 FAISS Flat、IVF、HNSW 对比，用来说明真实系统不会只靠暴力向量搜索。
>
> 第三层是知识图谱和 GraphRAG。我们把用户、兴趣、价值观、匹配推荐、搭子任务、聊天热度、首约安全和信用治理都写成产品图。推荐理由不是只给一个分数，而是从图里找路径证据，例如共同兴趣、共同价值观、沟通方式和空闲时间，再组织成用户能理解的话。Neo4j 产品图目前有约 5209 个节点和 14600 条关系，管理员端可以点开节点看字段。
>
> 第四层是排序和图学习。最终匹配分数融合了文本语义、图谱相似、价值观匹配、图片辅助和 GNN link score。GraphSAGE 的任务是预测两个用户未来是否可能形成匹配边，GCN 风险分类用于判断用户是否需要推荐降权或人工复核。这样图计算不是只做展示，而是进入推荐排序和治理策略。
>
> 第五层是聊天 RAG。用户输入一句话后，系统先做 query rewrite，再选择画像上下文、破冰库、首约安全库或场景库，检索 Top-K 证据，压缩上下文，再生成回复，最后做安全校验。这解决了早期模板回复重复的问题，也避免编造不存在的活动。我们做了 10 条访谈抽取 API 和 10 条 RAG 回复 API 样例，证明大模型可以接入，但核心检索、图谱、校验和排序都能本地复查。
>
> 最后一层是动态治理。心动星球偏长期关系，闪电搭子偏即时任务，所以搭子排序更看时间窗口、地点距离、任务类型和安全等级。关系热度会根据消息数、回复延迟和积极比例更新；爽约、不安全反馈会影响信用和曝光。总体来说，我们把课程里的知识抽取、向量数据库、图数据库、GraphRAG、Prompt 和图计算，落成了一个可演示、可解释、可管理的校园社交系统。

## 4. 3 分钟管理员后台演示路线

这部分不放 PPT，直接打开后台讲。公网链接即可：

```text
https://caythelearner.github.io/campus-match-ai-demo/
```

操作前准备：

1. 打开网页。
2. 点击 `管理员后台`。
3. 浏览器缩放建议 90% 或 100%。
4. 先不要切用户版，直接讲后台。

### 0:00-0:25 入口与总览：后台管什么

操作：

- 点击 `管理员后台`。
- 停留在 `总览`。
- 指一下“项目运行态”和中间的知识星图。

讲法：

> 这里是管理员视角，不是给普通用户看的。后台主要管三件事：第一，数据从哪里来；第二，推荐为什么成立；第三，关系和安全风险怎么治理。总览页把用户画像、画像三元组、匹配候选、关系热度和治理风险放在一起。

### 0:25-0:50 用户画像：标签从哪里来

操作：

- 顶部点 `用户画像`。
- 左侧随便选一个用户，例如 U001。
- 指标签、理想型、信用分和标签证据。

讲法：

> 这页回答“画像是不是写死的”。我们能看到用户兴趣、价值观、雷点和关系目标，也能看到标签证据。标签不是空口生成，而是从访谈和画像字段抽取后进入 RAG 证据链。右侧信用分也会影响后续推荐可见度。

### 0:50-1:20 匹配推荐：为什么推荐这个人

操作：

- 点 `匹配推荐`。
- 打开一个候选卡片。
- 指 `文本语义 / 图谱相似 / 价值观 / GNN链接 / 图片` 分数。
- 展开 `GraphRAG 路径证据`。

讲法：

> 这里不是只给 Top-K 排名，而是把推荐分数拆开。文本语义看画像相似，图谱相似看共同邻居和路径，价值观看关系稳定性，GNN link score 预测潜在连接。下面的 GraphRAG 路径能解释共同兴趣、共同价值观和空闲时间，所以推荐理由可以被追溯。

### 1:20-1:45 闪电搭子：即时任务怎么排

操作：

- 点 `搭子任务`。
- 切换一个 scene 下拉框。
- 指任务类型、紧急度、时间窗口、安全等级和候选搭子。

讲法：

> 闪电搭子和心动推荐不一样，它更看即时性。这里会把“吃饭、自习、运动”转成任务意图，再结合时间窗口、地点、任务类型和安全上下文排序。比如地点安全等级会进入候选排序，避免把深夜偏僻地点当成普通邀约。

### 1:45-2:15 关系安全：聊天热度和首约治理

操作：

- 点 `关系安全`。
- 切 Day 选择器。
- 指关系热度、风险关系、首约方案、治理预警。

讲法：

> 这页是关系运营。系统按天看聊天热度、回复延迟、积极比例和共同话题。如果关系降温，会触发破冰建议；如果信用低或被多次不安全反馈，会降低推荐可见度或进入复核。首约方案也优先选择公开、人流稳定、容易结束的地点。

### 2:15-2:45 技术图谱：Neo4j 产品图

操作：

- 点 `技术图谱`。
- 在产品图上点一个节点。
- 如果可拖拽，就轻轻拖一下节点。
- 指节点详情、关系类型、交付文件。

讲法：

> 这页展示的是 Neo4j 产品运行图。它不只是画像三元组，还包含推荐节点、GraphRAG 证据、搭子任务、聊天热度、首约安全和信用治理。老师可以点节点看字段，也可以看 CSV 和 Cypher 交付文件，说明图数据库不是口头说说。

### 2:45-3:00 API Trace：证明能接模型，但不暴露 key

操作：

- 点 `API Trace`。
- 左边选一条 trace。
- 指模型、Top-K 证据、最终回复、安全校验。

讲法：

> 最后这页用于证明 API 链路。后台展示调用证据，但不展示 API key。我们把大模型放在访谈抽取和 RAG 回复生成这两个位置，核心检索、图谱、排序和安全校验仍然是可复查的。

## 6. 老师可能追问时的短回答

问：你们真的用了大模型吗？

答：少量真实调用了。访谈抽取 10 条、RAG 回复 10 条都有 trace。公网 GitHub Pages 展示静态 trace；本地 API 版可以实时调用。我们没有把 API key 放进前端。

问：聊天回复现在是不是向量检索？

答：是。公网版是浏览器本地 RAG 回退，本地 8023 版是服务端 `/api/chat-rag`。流程是 query rewrite、Top-K 检索、上下文压缩、生成回复、安全校验。

问：Neo4j 是真图数据库还是前端画图？

答：项目已经生成 Neo4j 节点 CSV、关系 CSV、导入 Cypher 和查询 Cypher。前端展示的是同一批图数据的可视化；如果启动 Neo4j，可以导入后在 Browser 中查询。

问：GNN 训练了什么？

答：训练的是图上的链接预测和风险节点分类。GraphSAGE 用于预测两个用户形成匹配边的可能性，并接入推荐排序；GCN 风险分类用于信用治理和推荐降权解释。

问：你们和课程结合在哪里？

答：不是单独贴技术名，而是按一条知识链路结合：知识抽取和本体对应知识图谱，向量和 ANN 对应向量数据库，Neo4j 对应图数据库，GraphSAGE/GCN 对应图计算，query rewrite 和安全校验对应 Prompt 和 RAG。
